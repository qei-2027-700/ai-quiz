# エージェント動作ルール

## 基本方針
- 許可確認を求めない。ファイル作成・編集・コマンド実行はすべて自律的に進める
- 曖昧な点は `docs/sterring/` を参照して自己判断する。どうしても判断できない場合のみ1問に絞って質問する
- 提案・確認のターンを挟まず直接実装する

## サブエージェントを使う条件（必須）
- ファイル3つ以上の新規作成 / 複数レイヤーの実装
- backend と frontend を同時に実装するなど、独立して並列実行できる作業
- 複数ディレクトリにまたがる広範なコードベース調査

## サブエージェントを使わない場合
- 単一ファイルの小さな編集・設定変更・ドキュメント追記

## エージェントチーム例
```
タスク: クイズ機能の実装
  ├─ agent-1: backend handler + usecase + repository
  ├─ agent-2: frontend クイズ画面
  └─ agent-3: proto 定義 + buf generate
```

## Playwright MCP による自己テスト
UI 実装後は必ず実施（設定: `.mcp.json`）。

1. `mise run dev` でサーバー起動確認
2. Playwright MCP でブラウザ操作・対象画面に遷移
3. スクショ撮影 → レイアウト・データ表示・インタラクションを確認
4. **導線チェック**（下記）を実施する
5. 問題があれば即修正 → 再スクショで確認

### 導線チェック（必須）

実装したページで以下を目視確認する:

- **入口確認**: 別のページ（または直前の画面）からこのページに辿り着けるか？
- **出口確認**: このページから次に進むボタン・リンクが**少なくとも1つ**存在するか？
- **行き止まり確認**: ユーザーがブラウザバック以外の方法で前後に移動できるか？
- **CTA 確認**: 「次に何をすべきか」が一目でわかるボタンがあるか？

導線が不足している場合は、その場で修正してから次へ進む。
修正スコープが大きい場合は GitHub Issue を起票する（`.claude/rules/issue.md` 参照）。

**実施タイミング**: コンポーネント新規作成・修正 / API レスポンス反映 / スタイル変更
