# docs-sync

目的: `docs/sterring/` 配下のいずれかのドキュメントが変更されたとき、関連する他の md ファイルとの整合性を確認・修正する。

## 使う場面

- 設計ドキュメントを更新した直後
- 「docs を同期して」と指示されたとき
- 実装が完了してドキュメントが古くなったと気づいたとき

## 手順

### Step 1: 変更されたファイルを特定する

引数がある場合はそのファイルを対象にする。
引数がない場合は直近の git 差分から変更された md ファイルを特定する:

```bash
git diff --name-only HEAD~1 HEAD -- '*.md'
git diff --name-only -- '*.md'  # 未コミットの変更も確認
```

### Step 2: 関連ファイルを特定する

変更されたファイルに応じて、以下の対応表から関連ファイルを確認する:

| 変更ファイル | 確認すべき関連ファイル |
|------------|-------------------|
| `docs/sterring/tech-stack.md` | `CLAUDE.md`・`docs/sterring/development-environment.md` |
| `docs/sterring/auth.md` | `CLAUDE.md`・`docs/sterring/api-design.md` |
| `docs/sterring/api-design.md` | `docs/sterring/auth.md`・`proto/` の定義との整合 |
| `docs/sterring/ai-feedback.md` | `CLAUDE.md` の「現在のフェーズ」・`docs/sterring/tech-stack.md` |
| `CLAUDE.md` | `docs/sterring/` 全体 |
| `.claude/rules/*.md` | `CLAUDE.md` の「コーディング規約」セクション |

上記にない場合はファイルの内容を読み、どのドキュメントと関連するかを自己判断する。

### Step 3: 整合性を確認する

各関連ファイルを読み、以下の観点でチェックする:

- **矛盾**: 変更内容と相反する記述がないか
- **古い情報**: 変更前の状態を指している記述がないか
- **参照切れ**: 変更によってリンク先・ファイル名が変わっていないか
- **抜け漏れ**: 変更内容が他のドキュメントに反映されるべきなのに記載がないか

### Step 4: 修正する

不整合が見つかった場合は Edit ツールで修正する。
修正した場合は変更内容を箇条書きで報告する。
問題がなければ「✅ 整合性確認済み：変更不要」と報告する。

## 出力フォーマット

```
## docs-sync 結果

対象ファイル: <変更されたファイル>
確認したファイル: <関連ファイル一覧>

### 修正した内容
- <ファイル名>: <修正内容>

### 問題なし
- <ファイル名>: 整合性 OK
```
