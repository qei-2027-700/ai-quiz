指定された GitHub Pull Request をレビューしてください。

## 手順

1. 引数として PR 番号または URL が渡された場合はそれを使う。なければ `gh pr list` で一覧を表示してユーザーに選ばせる
2. 以下を**並行して**取得する
   - `gh pr diff <number>` — 差分
   - `gh pr view <number> --json reviews,commits` — 既存レビュー全件 + コミット一覧
3. **既存レビューを必ず読む（最重要）**
   - 過去に指摘された内容を把握する
   - 「修正済み」「訂正」「LGTM」などが記録されている指摘は **再指摘しない**
   - コミットメッセージに `fix:` が含まれ、対応する指摘が過去レビューにある場合は解決済みとみなす
4. 以下の観点でレビューを行い、`gh pr review` でコメントを投稿する

## must を使う前の必須チェック

**must（マージブロック）は以下の条件をすべて満たす場合のみ使う：**

- [ ] バグ・問題が実際に存在することを**コード上で確認できる**（推測・疑念ではない）
- [ ] ライブラリ・ランタイムの挙動に関する主張は、ソースコードまたは実行結果で裏付けできる
- [ ] 同じ内容が過去のレビューで指摘・対応済みでないことを確認した

上記を満たせない場合は want / nit / imo に格下げする。

## レビュー観点

### 共通
- CLAUDE.md・docs/sterring/ の規約に違反していないか
- 不要なファイル・デバッグコードが含まれていないか

### Backend (Go)
- Clean Architecture の依存方向が守られているか（domain ← usecase ← handler）
- sqlc の生成コードを直接編集していないか（queries/*.sql を変更すべき）
  - **注意**: `queries/` を変更して `gen/` も変わっている場合は `sqlc generate` による再生成とみなす（手動編集ではない）
- Connect-RPC のエラーが `connect.NewError` を使っているか
- SQL に N+1 の懸念がないか

### Frontend
- `@ai-quiz/api-client` / `@ai-quiz/shared` 経由で API を呼んでいるか
  - **注意**: `api-client` パッケージ内部での `fetch` 使用は許容（パッケージ外からの直接 fetch を禁止するルール）
- TailwindCSS クラスが可読性を保っているか

### IaC
- シークレットがハードコードされていないか
- Terraform の変更に `plan` 結果のコメントがあるか

## 重要度の定義

| 重要度 | 基準 |
|-------|------|
| **must** | マージするとバグ・セキュリティ問題が確実に発生する。コードで証明できる |
| **want** | 品質向上に有効だが、マージをブロックするほどではない |
| **nit** | スタイル・可読性の細かい改善 |
| **imo** | 設計上の意見。対応は任意 |

## レビューの出力形式

- 問題あり: `gh pr review <number> --comment -b "..."` でコメント投稿
- 承認: `gh pr review <number> --approve`
