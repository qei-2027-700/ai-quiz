# AI Issue-Driven Development

AI が自律的に課題を検出・起票し、それを参照して実装する半自動開発フロー。

---

## 全体フロー

```
【検出フェーズ】
  コードレビュー / lint / 実装中 / Playwright 自己テスト
        ↓ AI が改善点・TODO を発見
  GitHub Issue を自動起票（ラベル: ai-detected）

【判断フェーズ】
  ↓ 人間がラベル・優先度を確認（approve / close）

【実装フェーズ】
  "implement #X" / "/gh-impl #X"
        ↓ AI が Issue を読んで実装
  PR 作成 → CI → マージ
```

---

## Issue ラベル体系

| ラベル | 用途 |
|-------|------|
| `ai-detected` | AI が自動検出したタスク（必ず付ける） |
| `feat` | 新機能 |
| `fix` | バグ修正 |
| `refactor` | リファクタリング・コード品質 |
| `tech-debt` | 技術的負債 |
| `docs` | ドキュメント |
| `chore` | ビルド・CI・依存関係 |
| `priority:high` | 優先度高（セキュリティ・クラッシュ等） |
| `priority:low` | 優先度低（nice-to-have） |

---

## AI が Issue を起票すべきタイミング

### 自動起票（指示がなくても起票する）

- **セキュリティ上の問題**を発見したとき（SQL injection, XSS, 認証漏れ等）
- **クラッシュ・データ損失につながるバグ**を発見したとき
- `/claude-lint` の結果、修正できない or 修正範囲が大きい問題があるとき
- Playwright 自己テストで**再現性のある UI バグ**を発見したとき

### 提案起票（実装中に気づいたら起票）

- TODO / FIXME コメントが残っているとき
- 明らかな N+1 クエリ・パフォーマンス問題があるとき
- テストが存在しない重要なビジネスロジックがあるとき
- 型定義が `any` / `interface{}` で逃げているとき
- 設計ドキュメント（`docs/sterring/`）と実装が乖離しているとき

### 起票しない（その場で修正する）

- 変更が1〜2ファイル以内で完結する小さな修正
- 現在の作業スコープに含まれる変更

## Issue テンプレート

```markdown
## 概要
<!-- 何が問題か / 何を実装すべきか を1〜2行で -->

## 背景・理由
<!-- なぜこれが必要か -->

## 実装方針（案）
<!-- AI が提案する実装アプローチ。人間が上書きしてよい -->
- [ ] ...
- [ ] ...

## 影響範囲
<!-- 変更が及ぶファイル・レイヤー -->

## 参考
<!-- 関連 Issue・PR・ドキュメントへのリンク -->

---
🤖 Detected by Claude Code
```

## 実装フェーズの進め方

Issue を参照して実装する際は以下を守る:

1. `gh issue view <number>` で Issue の内容を読む
2. Issue の「実装方針」を起点にするが、コードを読んで最適化してよい
3. コミットメッセージに `Closes #<number>` を含める（自動クローズ）
4. PR の body にも `Closes #<number>` を書く

```bash
# 実装の起点
gh issue view 42

# コミット例
git commit -m "feat: ログイン済みの場合にログインボタンを非表示にする

Closes #42"
```

## 半自動化の範囲と人間の役割

| フェーズ | AI | 人間 |
|---------|-----|------|
| 検出・起票 | 自動実行 | レビュー・優先度付け・不要なら close |
| 実装 | 指示を受けて実行 | `implement #X` と伝えるだけ |
| PR レビュー | `/gh-rv` でセルフレビュー | 最終承認・マージ |
| リリース | CI/CD が自動実行 | タグ打ち（本番のみ手動承認） |

## 将来構想：完全自動ループ

現在は「人間が `implement #X` と指示する」ステップが必要だが、将来的には:

```
GitHub Actions（cron / webhook）
  → Claude Code をトリガー
  → `priority:high` な Issue を自動実装
  → PR 作成 → CI パス → 自動マージ（staging のみ）
```

この構想の詳細は `docs/sterring/development-environment.md` に記載。
