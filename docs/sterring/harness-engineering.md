# ハーネスエンジニアリングの観点

## 概要

ハーネスエンジニアリングとは、システムを「テスト・検証・観測がしやすい構造」で設計する考え方。
このプロジェクトでは以下の観点で設計の妥当性を判断する。

---

## チェックリスト

### 1. インターフェース駆動設計（Interface-driven）

- [ ] Clean Architecture の各層が interface 経由で依存しているか
  - `usecase` 層は `repository` の interface に依存し、具体実装には依存しない
  - handler のテストで `usecase` をモック可能か
- [ ] Connect-RPC の proto 定義が「テスト可能なコントラクト」になっているか
  - `buf breaking` で後退互換の破壊を CI で検知できるか

### 2. ローカル再現性（Local Reproducibility）

- [ ] `mise run setup` 一発でゼロから開発環境を構築できるか
- [ ] Docker Compose で全依存（DB 等）をローカルで起動できるか
- [ ] `mise run migrate-up` でスキーマを確定的に再現できるか
- [ ] シードデータ（初期クイズ問題）を投入するスクリプトがあるか

### 3. テスト分離（Test Isolation）

- [ ] ユニットテスト: 外部依存なし（DB・API 呼び出しなし）で実行できるか
- [ ] インテグレーションテスト: テスト用 DB（Docker）に対して実行できるか
- [ ] E2E テスト: Playwright MCP でローカル環境に対して実行できるか
- [ ] 各テストが独立して実行でき、順序依存がないか

### 4. 観測可能性（Observability）

- [ ] 構造化ログ（`go.uber.org/zap`）が適切に出力されるか
- [ ] `GET /healthz` ヘルスチェックエンドポイントが存在するか
- [ ] エラーレスポンスに十分なコンテキスト情報が含まれるか
- [ ] Connect-RPC のエラーコードが適切に分類されているか

### 5. 契約テスト（Contract Testing）

- [ ] proto 定義が API の「唯一の真実」として機能しているか
- [ ] `buf lint` で命名・スタイル規約が守られているか
- [ ] `buf breaking` で非互換変更が検知されるか（CI で実施）
- [ ] sqlc の `sqlc vet` で SQL の型安全性が検証されているか

### 6. 冪等性（Idempotency）

- [ ] マイグレーションは何度実行しても同じ結果になるか（`up/down` が対称）
- [ ] シードデータの投入スクリプトは冪等か（二重投入しても壊れないか）
- [ ] Terraform の `apply` は冪等か（変更がない場合は何もしないか）

---

## 実装時の判断基準

新しいコードを書くとき、以下を確認する:

```
このコードはテストできるか？
  → できない場合は interface を切って依存を注入できる構造にする

このコードはローカルで動かせるか？
  → 外部サービス依存がある場合はモック or docker-compose にサービスを追加する

このコードが壊れたとき、すぐ気づけるか？
  → ログ・エラーコード・ヘルスチェックで観測できる構造にする
```
